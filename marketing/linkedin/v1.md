# CrispyBrain LinkedIn Post (v1)

I've been building CrispyBrain as a local-first memory and retrieval lab.

The point is not to present a finished AI product. The point is to make the retrieval path visible: what got stored, what got retrieved, and how much trust to place in the answer that comes back.

The current setup is intentionally real:

- a local demo UI
- checked-in n8n workflows
- self-hosted runtime
- plain-text ingest into project memory
- source and trust metadata in responses

In the simplest path today, you drop a `.txt` file into a project inbox, wait for the ingest/watch flow, then ask a question against that project and inspect the result.

That is useful to me because it turns "AI memory" from a vague promise into something observable.

It is also very clearly still early:

- not production-ready
- not turnkey
- still manual in a few important places
- strongest in the current local demo path, not as a general platform

I think there is a lot of value in building these systems in a way that makes the workflow legible instead of magical.
